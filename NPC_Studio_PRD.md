# NPC Studio — 产品需求文档（PRD）

> **项目背景**：LIF001 Group D30 — AI in Gaming and Virtual Reality（学年 2025-26 第二学期 Assessment 2 产品提交）。
>
> **文档定位**：本文档只描述"做什么"和"为什么"，不包含具体代码实现。开发者（包括 AI 编码助手）应在此 PRD 基础上自行选择代码实现方式。
>
> **目标读者**：本组成员、负责开发的同学、负责视觉设计与文案的同学、以及任何受邀协作者。

## 1. 产品概念

**NPC Studio** 是一个面向游戏设计师、玩家与研究者的网页应用。用户在页面上配置一个 AI 驱动的 NPC（非玩家角色）的**性格设定、角色等级、情感强度和伦理保护开关**，然后即时与该 NPC 对话，亲身感受不同配置下 NPC 表现的差异。

产品的核心价值是**把第一学期研究中提出的四条 NPC 设计主张做成可交互的实物**，让评分老师不用读论文，只需点击几下、和 NPC 聊几句，就能直接感受到我们方案的效果。

### 与第一学期研究的对应关系

| 第一学期提出的设计主张 | 在 NPC Studio 中的呈现方式 |
|---|---|
| **行为规则锁**（Behavioral Rule Lock）—— 主线 NPC 必须严格守约，路人 NPC 则有更大自由度 | 用户可在界面上选择"主线 NPC / 支线 NPC / 环境 NPC"三档，系统自动注入对应的行为约束 |
| **三档情感强度**（Low / Medium / High）—— 满足不同玩家对情感深度的偏好 | 滑块调节情感强度，实时改变 NPC 的回复温度、长度和情绪表达浓度 |
| **伦理与心理健康保护机制** —— 防止玩家过度情感依赖与虚实混淆 | 可开启"伦理模式"，触发条件下 NPC 会温和地打破第四面墙；累计使用 15 分钟自动弹出休息提醒 |
| **鼓励真人协作而非孤立沉浸** —— 减少对单一 NPC 的依赖 | 提供"分享配置链接"功能，鼓励玩家把当前 NPC 配置发给朋友，共同体验 |

### 产品形态归类

按 LIF001 任务书的产品分类，本产品同时具有：
- **多媒体创作类**（Multimedia Creation）的特征：交互式网页、即时 AI 对话；
- **实物功能类**（Physical Functional）的特征：可作为 NPC 设计方法论的参考方案。

提交方式：以**可访问链接**的形式提交（任务书允许的三种提交路径之一），无需打包为 PDF。

## 2. 项目目标（对照 LIF001 评分标准）

LIF001 Assessment 2 的评分标准共五项，每项 20%。本产品在每一项上的设计意图如下：

### 2.1 Problem Clarity（问题清晰度）20%
- 落地页第一屏直接陈述行业痛点：传统 NPC 对话同质化、缺乏行为约束、易引发玩家过度情感依赖；
- 痛点旁附简短的研究依据（团队后续填入 3 条引用）；
- 在主交互页提供"对照模式"开关，用户可以一键对比"未启用规则锁"与"启用规则锁"的差异，让问题在交互中被感知。

### 2.2 Solution Effectiveness（解决方案有效性）20%
- 四条设计主张全部做成可交互控件，而不是文字说明；
- 用户调整任何设置后，10 秒内必须能在 NPC 回复中感受到差异；
- 提供"行为日志"面板，把当前生效的规则用色块清晰列出。

### 2.3 Creativity & Innovation（创新性）20%
- 把"NPC 设计原则"从论文里搬到可玩的 demo —— 这是绝大多数同年级小组不会做到的层级；
- 提供"系统提示词差异查看器"：用户可以看到自己的配置如何被翻译成发给大模型的 system prompt，体现技术透明度；
- 提供"分享配置链接"：通过 URL 编码把整套 NPC 配置传给朋友。

### 2.4 Presentation & Communication（呈现与沟通）20%
- 用现代 UI 组件库构建简洁专业的视觉；
- 桌面端三栏布局，移动端自适应（方便录制视频时镜头切换）；
- 完善空状态、加载状态、错误状态。

### 2.5 Reflection & Learning（反思与学习）20%
- 设独立的"关于"页，按 LIF001 学术诚信要求**完整披露 AI 工具使用情况**；
- 在该页坦诚记录团队踩过的坑：哪些是 AI 加速的、哪些是 AI 出错被人工纠正的；
- 提交视频结尾用 1 分钟讲反思，呼应"关于"页内容。

## 3. 核心设计原则（产品的灵魂）

四条原则在产品中**必须可被用户感知到**，而不是只在文档里描述。

### 3.1 行为规则锁（Behavioral Rule Lock）

**问题**：当前游戏 NPC 要么完全按脚本走（无趣），要么完全自由发挥（剧情容易崩）。

**我们的方案**：把 NPC 按"对剧情的关键程度"分三档，每档用不同的提示词约束。

| 角色等级 | 设计意图 | NPC 行为表现 |
|---|---|---|
| **主线 NPC**（Main Quest）| 必须严格守约 | 不会拒绝推动剧情的请求；不会脱离世界观；不可"罢工" |
| **支线 NPC**（Side Quest）| 适度自由 | 可以拒绝小请求，但整体仍配合玩家；保持世界观一致 |
| **环境 NPC**（Ambient）| 完全自由 | 可以走开、不理人、说与剧情无关的内容；唯一约束是自身性格一致 |

**附加开关**：当选择"主线 NPC"时，自动锁定"NPC 不可拒绝"的开关；用户也可手动覆盖。

### 3.2 三档情感强度（Three-Tier Emotion Intensity）

**问题**：不同玩家对 NPC 的情感深度需求不同 —— 有人只想快速领任务，有人想沉浸。

**我们的方案**：

| 强度档位 | 回复风格 | 长度上限 |
|---|---|---|
| **Low** | 功能性回复，只确认请求、给信息或任务，不带情感修饰 | 短 |
| **Medium**（默认）| 角色化回复，带 1 处情感小波动（一个回忆、一句反问）| 中 |
| **High** | 富情感纹理，含内心反应、犹豫、温度变化和邀请性的反问 | 长 |

技术上由**调节大模型温度参数 + 提示词中的情感指令 + 输出长度上限**三层共同实现。

### 3.3 伦理与心理健康保护（Ethics & Wellbeing Safeguards）

**问题**：第一学期数据收集中我们发现，过度真实的 NPC 可能引发玩家情感依赖、虚实混淆、伦理困境。

**我们的方案分三层**：

1. **伦理模式开关**（默认开启）：开启后，若玩家在对话中表现出真实世界的痛苦、孤立或对 NPC 的过度情感依赖，NPC 会**温和地打破第四面墙**一次（例如："朋友，这是游戏，外面的世界也在想你"），然后回到角色；
2. **会话时长提醒**：累计使用 15 分钟，自动弹出休息提醒模态框，提供"我去休息"和"再聊 5 分钟"两个选项；
3. **明确边界**：NPC 永远不会自称真人，不会指导玩家进行任何自伤或物质相关的行为。

### 3.4 鼓励真人协作（Co-op over Dependence）

**问题**：避免玩家把 NPC 当成唯一的情感寄托。

**我们的方案**：每个 NPC 配置都可以一键编码到 URL 中分享给朋友，朋友打开链接后看到的是同一套 NPC 配置 —— 鼓励两人或多人围绕同一 NPC 体验、讨论、调整，把单人沉浸引导为社交行为。

## 4. 用户角色与使用场景

### 主要用户

1. **游戏设计师 / 学生开发者**：想了解"AI NPC 该如何被有原则地设计"；
2. **玩家**：想体验 AI 驱动的 NPC 在不同设定下的差异；
3. **LIF001 评分老师**：在评审视频中或自行打开链接体验。

### 典型使用流程

1. 用户打开网站 → 落地页快速理解我们要解决什么问题；
2. 点击"试用 Studio"进入主交互页；
3. 从 10 个预设 NPC 中选一个（或保留默认）；
4. 调整角色等级 / 情感强度 / 伦理开关；
5. 输入消息，看 NPC 即时流式回复；
6. 修改配置后再发一句，亲身感受差异；
7. 点击"分享"复制链接发给朋友；
8. 累计使用 15 分钟时收到休息提醒；
9. 跳转"关于"页阅读 AI 使用披露与反思。

## 5. 功能模块

产品由三个页面 + 若干组件构成。

### 5.1 落地页（Landing）

承担"3 秒讲清楚问题、再用 30 秒讲清楚方案"的任务。

**页面分区：**

- **Hero 区**：产品标题、一句话副标题（"设计有原则的 AI NPC"）、主操作按钮"试用 Studio"、次按钮"查看 GitHub 源码"；
- **问题陈述区**：三张痛点卡片（NPC 同质化 / 缺乏约束 / 过度依赖风险），每张卡片附一条研究引用占位；
- **方案预览区**：四张方案卡片，对应第 3 节四条原则；
- **页脚**：链接到 Studio、关于页、GitHub。

### 5.2 主交互页（Studio）—— 产品核心

桌面端三栏布局，移动端自上而下堆叠。

#### 左栏（约 320px 宽）：配置区

- **NPC 选择器**：从 10 个预设 NPC 中切换，显示头像与简介；
- **配置面板**：
  - 角色等级单选（主线 / 支线 / 环境）
  - 情感强度滑块（低 / 中 / 高 三档）
  - 伦理模式开关（默认开启）
  - "允许 NPC 拒绝"开关（角色等级为主线时自动关闭并附说明）
  - "重置为默认"按钮

#### 中栏（自适应宽）：聊天区

- 上方：对话历史（玩家与 NPC 的消息按对话气泡呈现，配色区分）
- 中部：流式输出指示（NPC 思考时显示打字光标）
- 下方：输入框（textarea + 回车发送 + 发送按钮，发送中禁用）
- 工具：清空对话按钮（保留 NPC 配置不重置）

#### 右栏（约 320px 宽）：行为日志

- **当前生效规则**：以彩色 chip 列出（例如"主线锁定 / 情感：中 / 伦理：开"）
- **可折叠面板**："查看完整系统提示词" —— 展示当前配置生成的完整 system prompt，体现技术透明
- **小字说明**："这些规则通过提示词传递给大模型，并非硬编码强制约束"

#### 顶栏

- **场景选择器**：8 个起步场景下拉，选中后自动填入起手消息
- **分享按钮**：复制带配置的 URL 到剪贴板
- **会话时长徽标**：显示当前会话累计时长

### 5.3 关于页（About）—— AI 使用披露

按 LIF001 学术诚信要求，**必填**三部分：

1. **团队介绍**：D30 全体成员姓名、专业、分工（队长统筹、开发、内容、视觉、视频、QA 等）；
2. **AI 工具使用清单**：
   - Claude Code（代码生成与脚手架）
   - Claude Sonnet API（产品运行时的 NPC 对话生成）
   - 其他实际使用的工具（团队补充）
3. **反思段落**（约 200-300 字）：AI 加速了什么、AI 出错了什么、人类做了什么验证。

## 8. 提示词构建逻辑（产品的核心 IP）

整个产品的"智能"集中体现在**一个纯函数**：输入 = (NPC 性格 + 用户配置)，输出 = 发给大模型的完整 system prompt + 调用参数。

这个函数是**人写的、可解释的、可被审查的**，不依赖大模型自身的"理解"。

### 提示词的 6 层结构

每次请求拼装的 system prompt 按以下顺序叠加 6 层：

| 层级 | 内容 | 由谁决定 |
|---|---|---|
| **第 1 层** | NPC 性格核心（名字、原型、背景、说话方式）| NPC 库（§6）|
| **第 2 层** | 行为规则锁（主线 / 支线 / 环境 三选一的约束）| 用户在配置面板的选择 |
| **第 3 层** | 拒绝覆盖（"不可拒绝主线请求"）| 用户开关 |
| **第 4 层** | 情感强度指令（低 / 中 / 高 的具体描述）| 用户滑块 |
| **第 5 层** | 伦理保护指令（仅在伦理模式开启时注入）| 用户开关 |
| **第 6 层** | 世界观提醒（短句，作为最后一道场景锚定）| NPC 库（§6）|

### 调用参数的同步调整

不只是改提示词，配置同时影响：
- **温度**（temperature）：低情感档 0.3 / 中档 0.6 / 高档 0.85；
- **最大输出 token**：低 ≈ 160 / 中 ≈ 320 / 高 ≈ 480；

确保情感强度的差异**用户能直观感受**到（短/中/长 + 平淡/中庸/丰富）。

### 透明度

每次请求时，**生效的规则列表**会随同回复一起返回给前端，由"行为日志"组件展示。用户随时可以折叠展开看到完整的拼装好的 system prompt —— 这本身就是一种创新点（绝大多数 AI 产品对用户隐藏 system prompt）。

## 9. 技术栈选型（高层方向，不锁死实现）

| 层级 | 推荐选型 | 选择理由 |
|---|---|---|
| **前端框架** | Next.js 14（App Router）+ TypeScript | 生态成熟、AI 编码工具最熟悉、Vercel 一键部署 |
| **样式** | Tailwind CSS + shadcn/ui | 开箱即用的精美组件、改动成本低 |
| **状态管理** | Zustand（全局）+ React 内置（局部）| 学习成本几乎为零，够用 |
| **大模型 API** | Anthropic Claude Sonnet（主选）；DeepSeek 或通义（备选）| 主选效果最好；备选解决国内访问与费用问题 |
| **流式传输** | Server-Sent Events（SSE）| 标准浏览器 API，比 WebSocket 简单 |
| **持久化** | 浏览器 localStorage | 无需后端数据库，零运维 |
| **部署** | Vercel | 免费、有自定义域名、生成可分享链接 |
| **代码托管** | GitHub（公开 repo）| 也作为反思报告的过程证据 |

> **替换原则**：以上选型可在不影响产品行为的前提下替换。如果团队成员更熟悉某些工具（如 Vue / Svelte / Vite），可以替换前端框架，但需在反思报告中说明替换理由。

## 10. 视觉设计要求

### 色彩
- **主色**：暖暖的羊皮纸/参天古树色调（贴合"奇幻 NPC"主题）；
- **次色**：石板灰，用于玩家消息气泡；
- **强调色**：1 个鲜明色（橙或青），用于按钮和当前生效规则的 chip；
- **支持暗色模式**（响应系统设置）。

### 排版
- 中文字体优先：思源黑体 / 苹方 / 系统默认；
- 英文字体：Inter 或系统 sans-serif；
- 标题加粗大号，正文 16px 起，行高 1.6。

### 状态完整性
- 空状态（"还没有 NPC 在等你 —— 选一个开始吧"）；
- 加载状态（骨架屏 + 流式输出光标）；
- 错误状态（"NPC 走神了，再试一次？"+ 重试按钮）。

### 移动适配
- 三栏布局在 < 1024px 宽时变为单栏垂直堆叠；
- 配置区折叠为可下拉抽屉；
- 行为日志折叠为底部 sheet。

## 11. 数据流与交互流程

### 11.1 单次发消息的完整链路

1. 用户在输入框输入消息，回车 / 点击发送；
2. 前端把"NPC ID + 当前配置 + 历史对话 + 新消息"打包发给后端 API；
3. 后端读取对应 NPC 设定，调用提示词构建函数生成 6 层 system prompt + 参数；
4. 后端调用大模型 API，开启流式输出；
5. 后端先回传一个事件告诉前端"当前生效的规则列表"；
6. 后端把大模型的 token 一边接收一边转发给前端；
7. 前端边接收边把 token 拼进 NPC 的回复气泡（打字机效果）；
8. 流式结束后，前端把整轮对话存进 localStorage。

### 11.2 配置分享链路

1. 用户调整完配置后点击"分享"；
2. 前端把当前 `(NPC ID + 配置)` 编码成 base64 字符串放到 URL 锚点（hash）；
3. 复制完整 URL 到剪贴板，弹出 toast 确认；
4. 朋友打开链接时，主交互页读取 hash，自动还原配置。

### 11.3 会话时长追踪

- 进入主交互页即开始累计本次会话用时；
- 用 localStorage 持久化，避免刷新丢失；
- 累计达 15 分钟弹出休息提醒模态框；
- 用户选"再聊 5 分钟"则把计时器重置为 10 分钟。
